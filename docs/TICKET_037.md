# TICKET-037 Implement: Frontend Onboarding Flow

다음 가이드라인에 따라 식물 등록 온보딩 UI 흐름을 구현해줘. 전체 문서를 읽지 말고 이 요약에만 집중해.

### 1. 목표 및 핵심 로직
- **목표**: 이미지 선택(Mock)부터 식물 등록 완료까지의 4단계 온보딩 프로세스 구현.
- **핵심 API 연동**:
  1. `POST /plants/species-candidates`: 선택한 이미지(`image_ref`)를 보내 식물 종 후보군 수신.
  2. `POST /plants`: 확정된 `species_id`, `nickname`, `room` 정보를 보내 식물 생성.

### 2. 온보딩 시퀀스 (User Flow)
1. **식물 추가 화면**: 카메라/갤러리 플레이스홀더 클릭 시 데모용 `image_ref` 선택.
2. **종 후보 확인 화면**: 백엔드에서 받은 후보 리스트(이름, 점수)를 보여주고 사용자에게 하나를 선택받음.
3. **프로필 설정 화면**: 식물의 별명(Nickname)과 키울 장소(Room - 거실, 침실 등) 입력.
4. **등록 완료 화면**: 생성된 식물 카드(별명, 종 이름, 장소)를 보여주며 홈으로 이동 유도.

### 3. 필수 구현 대상
- **Multi-step Form**: 각 단계의 상태(Step)를 관리하고 '이전/다음' 이동이 가능한 컴포넌트.
- **API Integration**: `X-User-Id` 헤더를 포함하여 후보군 요청 및 식물 생성 요청 수행.
- **UI Components**: 이미지 선택기(Mock), 후보군 카드 리스트, 입력 폼, 성공 결과 카드.

### 4. 제약 사항 (절대 금지)
- **실제 기능 금지**: 실제 카메라 캡처나 실제 파일 업로드 로직을 구현하지 마. (Mock `image_ref`만 사용)
- **백엔드 수정 금지**: 스키마 변경이나 API 수정을 절대 하지 마.
- **진단 기능 금지**: 온보딩 과정에서 병충해 진단이나 건강 체크 UI를 포함하지 마.
- **인증 UI 금지**: 회원가입이나 로그인 화면을 만들지 마. (T-036의 공통 헤더 정책 유지)

지금 바로 `src/pages/Onboarding/` 내의 단계별 컴포넌트 작성을 시작해줘.
