# Ticket 31 — Frontend Chat + Answer View

## 1. Goal

Implement the frontend chat screen so the user can ask plant-care, pest-reference, and companion-plant questions, then view the backend's fixed-format answer.

Ticket 31 is a **frontend rendering and API-call ticket**.
It must not change backend chat behavior, prompt logic, LLM behavior, persistence, Docker, or database schema.

## 7. API Client Contract

Add or extend:

```typescript
askChat(
  plantId: string,
  input: {
    question: string;
    locale: "ko-KR";
    request_id?: string;
  },
): Promise<ChatAnswerResponse>;
```

Required request:

```text
POST /plants/{plant_id}/chat
X-User-Id: demo-user-001
Content-Type: application/json
```

Required payload:

```json
{
  "question": "물 줘야 해?",
  "locale": "ko-KR"
}
```

`request_id` may be omitted and generated by the backend.
If the frontend generates it, tests must not depend on an exact runtime random value.

Forbidden frontend endpoint calls in this ticket:

```text
GET /chat-runs/{request_id}/evidence
GET /plants/{plant_id}/history
POST /plants/{plant_id}/care-logs
GET /plants/{plant_id}/care-logs

```

## 8. Fixed Answer Rendering Contract

The backend answer has four required sections:

```text
결론
근거
행동
주의
```

Frontend must render in this order:

```text
1. 결론
2. 근거
3. 행동
4. 주의
```

Required behavior:

```text
- prefer response.answer.sections
- render exactly four sections when sections exist
- if sections are missing but answer.text exists, parse fallback is allowed only when all four markers exist
- if any section is missing, show ChatErrorState
- do not fabricate missing section content
```

Forbidden:

```text
- merging all sections into one paragraph
- hiding 주의
- generating replacement advice locally
- changing backend care action based on frontend logic
```
